# Governance

## Model
Founder-led, community-supported.

This means:
- Community feedback is welcome and valued.
- The project maintainer is responsible for product direction and final decisions.
- Forks are allowed (that’s the point of open source), but the upstream repo stays coherent.

## Roles
### Project Lead (Maintainer)
- Sets roadmap and priorities
- Approves/rejects PRs
- Publishes releases
- Handles moderation decisions

### Contributors
- Open issues, PRs, docs improvements
- Propose features and discuss tradeoffs

## How decisions are made
- Small fixes: PR review + merge by maintainer
- Medium changes: issue discussion → PR
- Major changes: “RFC-style” issue discussion before any big build-out

If a feature conflicts with the project’s goals, it may be declined even if it’s technically correct.

## Community expectations
- Be respectful
- Assume good intent
- Keep discussions focused on the work

(If we add a formal Code of Conduct later, it will live at `CODE_OF_CONDUCT.md`.)

## Funding and content
- The software remains free and open under AGPL.
- Teaching/instruction (“Hour 2”) is separate content hosted on Wix.
- Optional funding rails (e.g., GitHub Sponsors) help sustain maintenance work.
